# deep-search-loop v5.5

> v5.5：Loop 增强版。三闸门质量控制——Gate A(搜索计划盲点检查) → Gate B(每轮质量不达标自动续轮) → Gate C(素材不够回 Step 2 补搜)。抓取层(Step 3)不变。v5.4 降级链铁律保留。

## Loop Control 架构

```
Step 1 ──→ Gate A ──→ Step 2(R1~R6) ──→ Gate C ──→ Step 4 → Step 5
 分析主题      搜索计划       每轮末尾        素材充分度    合成报告   交付
              盲点检查       Gate B 质检      不达标回
                            不达标自动续轮    Step 2
```

**不改动的（继承 v5.4）：**
- Step 0 保活检查
- Step 3 六层抓取 + 站点分层表
- Step 5 交付
- 安全规则、禁止事项、参考文件

---

## 核心原则

- **主力搜索：SearXNG → baidu JSON API**。用 `execute_code` 调 urllib 直连 `127.0.0.1:8880`，不走 terminal 避免审批。每轮走中文 + 英文两路搜索，结果合并去重。
- **英文补充：arXiv API**。`execute_code` + urllib 直连 `arxiv.org`。
- **停用后端不纠结**：ddgs / web_search / cn.bing.com / 360search / sogou / bing 全部不可用，不要试。Firecrawl 可用 keyless 模式（`FirecrawlClient()`，无需 API key）。
- **输出是阐述报告**：结构化分析，有数据有对比有判断有出处。不是 dump。
- **🔥 降级链铁律（v5.4 保留）**：每一篇 URL 必须走完降级链。L1 失败 → L2 → L2.5 → L3 → L4。**禁止"L1 失败就跳过"**。已知站点查 `references/site-layer-map.md` 直接跳对应层，未知站点逐层试到底。每轮至少抓取 5 篇正文，不够就继续降级。
- **🔄 Loop 铁律（v5.5 新增）**：三个闸门必须逐道通过。任一道不达标 → 自动回环 → 不交付半成品。

## Gate A：搜索计划盲点检查（Step 1→2 之间）

在制定完 6 个搜索角度后，**必须先跑盲点检查**再进入 Step 2：

| 检查项 | 不通过则 |
|:--|:--|
| 6 个角度是否覆盖了用户问题中所有关键词？ | 补充缺失角度 |
| 是否有角度过于抽象、无法转化为具体搜索词？ | 拆成 2-3 个子角度 |
| 参考 `references/mlcp-case.md`，类似主题上次踩过什么坑？ | 在计划中标注"已知陷阱" |
| 是否需要 arXiv 学术支撑？如果需要，是否预留了英文搜索词？ | 补齐英文关键词 |

**盲点检查输出格式：**
```
=== Gate A 盲点检查 ===
✅ 6 角度覆盖主题全部维度
✅ 所有角度有具体搜索词
⚠️ 已知陷阱：这个主题百度常返回百科类低质结果 → R1 搜索词加 "-百科 -文库"
✅ 学术支撑需要 arXiv（预留 3 个英文搜索词）
→ Gate A 通过，进入 Step 2
```

## Step 2：执行搜索 + Gate B 每轮质检

### 搜索流程

```
每轮流程（用 execute_code 一次性执行）：
  ① 确定本轮搜索词（R1 用 Step 1 预设词；R2+ 从上轮结果提炼）
  ② SearXNG 百度中文 + 百度英文双路搜索
  ③ 去重合并 → 筛选有价值 URL
  ④ 抓取正文（Step 3 降级链）→ 提取关键信息
  ⑤ Gate B 质检（见下）
  ⑥ Gate B 通过 → 可终止。不通过 → 自动续轮，调整搜索词
```

**R1-R3：强制执行，不做终止判断。R4-R6：任一触发即可停。**

### Gate B：每轮质量闸（每轮末尾强制执行）

### 深度判定标准（四维实质判据）

**不是数模块数量，是审内容质量。** 每轮末尾逐角度对照四维判据打分：

| 判据 | 浅（1分） | 中（2分） | 深（3分） |
|:--|:--|:--|:--|
| **数据密度** | 泛泛描述，无数字 | 有数字但无出处 | 有数字 + 出处 + 多源交叉验证 |
| **时效性** | 2024 年及以前 | 2025 年 | 2026 年且标注具体月份/事件 |
| **颗粒度** | "市场很大""需求旺盛" | "71 亿美元" | "71 亿，FC-BGA 占 34%，CAGR 25.3%，驱动因素是..." |
| **因果链** | "因为 AI 所以需要" | 点到具体瓶颈（如"翘曲导致良率低"） | 瓶颈→材料参数差异→工程方案→落地节奏，完整因果链 |

**打分规则：**
- 四维均 ≥2 分 → 该角度为"中"（及格）
- 四维均 ≥2 分 + 至少两维 = 3 分 → 该角度为"深"（优良）
- 任一维 = 1 分 → 该角度为"浅"（不达标），**不因其他维高分而中和**

**反作弊规则 —— 杜绝"数模块凑数"：**
- ❌ 严禁：看到角度有文章覆盖就标"深"
- ❌ 严禁：同一篇低质文章同时充当多个角度的"深"——每角度必须有独立支撑
- ❌ 严禁：用产业/金融类文章充当技术原理角度——技术角度必须源自技术文档/学术论文/工程师博客
- ❌ 严禁：把 2024 年的预测当 2026 年的"进展"——进展角度必须引用 2025-2026 的实际事件

### Gate B 清单格式

每轮抓取完正文后，**必须填写**：

```
=== Gate B 质检 — 第 N 轮 ===

一、逐角度四维评分
角度1(定义概述):
  数据:2 时效:3 粒度:2 因果:2 → 中 [简述已获取的核心信息]
角度2(原理技术):
  数据:1 时效:2 粒度:2 因果:2 → 浅 ❌ [工控网科普文，缺学术级工艺参数/良率数据]
角度3(应用场景):
  数据:1 时效:1 粒度:1 因果:2 → 浅 ❌ [腾讯文章2024年，无具体产品型号/客户案例]
角度4(产业格局):
  数据:2 时效:3 粒度:2 因果:2 → 中 [有A股名单但缺产能规划/订单金额]
角度5(进展动态):
  数据:1 时效:3 粒度:1 因果:2 → 浅 ❌ [仅"2026元年"一句，无具体送样客户/验证进度]
角度6(竞争挑战):
  数据:3 时效:3 粒度:3 因果:3 → 深 ✅ [ABF/硅/玻璃三材料参数对比+工程方案+落地节奏]

二、来源多样性
域名数: N / 已抓取正文 ≥500字: N 篇 / 独立支撑角度数: N(每角度至少1篇独立)
主流媒体: N  行业垂直: N  技术社区/学术: N  社交: N

三、本轮判定
浅层(任一维=1分): N 个角度  → 阈值: 浅层=0 且 深层≥3
深层(全维≥2+≥两维=3): N 个角度
→ Gate B [通过/不通过]
→ 不通过时: 下轮侧重 <列举不达标角度及具体改进策略>
```

**通过条件（必须全部满足）：**
- 浅层 = 0（六个角度全部至少"中"——四维全 ≥2）
- 深层 ≥ 3（至少三个角度达到"深"——全维 ≥2 + ≥两维 =3）
- 每角度至少有一篇独立来源支撑
- 来源域名 ≥ 3 个不同域名

**不通过对策（必须给出具体改进方案，不能只写"继续深挖"）：**
- 数据密度低 → 搜"<主题> 数据 统计 报告 2026"或搜学术论文/行研报告
- 时效性差 → 搜"<主题> 2026"或搜新闻/公告/季度报告
- 颗粒度差 → 搜细分关键词（如"玻璃基板 激光钻孔 孔径 深宽比 参数"）
- 因果链弱 → 搜"<主题> 为什么 原因 瓶颈 挑战"
- 域名单一 → 从缺失域名的候补 URL 中补抓
- 正文不足 → 翻页（page=2）或换搜索引擎

**硬上限：** R6 仍未通过 Gate B → 输出已有结果 + 标注每个不达标角度的具体缺口 + 建议手动补搜方向

## Step 3：抓取正文（继承 v5.4，完全不变）

搜到的有价值 URL 需要读正文。按以下优先级分层：

| 优先级 | 工具 | 适用 |
|:--:|------|------|
| 1 | execute_code + urllib | 新闻/博客/CSDN/知乎 |
| 2 | scrapling | urllib 返回空白/轻反爬 |
| 2.5 | Botasaurus @browser | Cloudflare/JS渲染 |
| 3 | browser_navigate | 微信/东方财富 |
| 4 | arXiv API | 学术论文 |
| 5 | Firecrawl scrape | 前5层全挂的关键URL |
| 6 | browser-act | 手动触发 |

## Gate C：素材充分度闸（Step 3→4 之间）

Step 3 全部抓取完毕后，**必须先过 Gate C 再写报告**：

```
=== Gate C 素材充分度检查 ===

对报告所需的 6 个章节逐一验证：

1.是什么:   素材数3  最佳来源: [URL]  深度: 深 ✅
2.原理/技术: 素材数2  最佳来源: [URL]  深度: 中 ⚠️ [缺具体参数对比表]
3.应用场景:  素材数1  最佳来源: [URL]  深度: 浅 ❌ [仅泛泛介绍]
4.产业格局:  素材数2  最佳来源: [URL]  深度: 中 ✅
5.进展/动态: 素材数2  最佳来源: [URL]  深度: 深 ✅
6.竞争/挑战: 素材数1  最佳来源: [URL]  深度: 浅 ❌ [无替代方案对比]

总素材: 11篇  ≥500字: 8篇  域名: 4个

浅层/无素材章节: 第3章(应用场景)、第6章(竞争与挑战)
→ Gate C 不通过 → 回 Step 2，针对缺失章节补搜

补搜方向:
  - "XX 应用场景 典型案例" → 倾斜角度3
  - "XX 替代方案 对比" → 倾斜角度6
  补搜轮次上限: 2 轮（R5-R6 名额内）
```

**通过条件：**
- 所有 6 章均 ≥ 中层覆盖
- 至少 3 章达到深层覆盖
- 正文 ≥500字的素材 ≥ 5 篇

**不通过对策：**
- 回 Step 2 执行补搜——仅搜索缺失章节对应的角度
- 最多补搜 2 轮（占用 R5/R6 额度）
- 补搜后重新过 Gate C
- 2 轮补搜仍不达标 → 在报告对应章节标注"⚠️ 浅层覆盖，数据有限"

## 安全规则

| 规则 | 说明 |
|------|------|
| **🔥 轮间间隔（v5.5.1 强化）** | **每轮之间 ≥30s**。本轮搜索+抓取全部完成后开始计时。宁可慢一轮，不能被封。被封 = 整条搜索链断裂 |
| **🔥 轮内查询串行（v5.5.1 新增）** | **禁止多路并行搜索**。同一轮内的多个搜索词必须逐个串行，间隔 ≥8s |
| **🔥 搜+抓总频控（v5.5.1 新增）** | 搜索请求和正文抓取走同一 SearXNG 通道。每轮搜+抓 HTTP 请求总数 ≤12 次 |
| **CAPTCHA 检测** | 百度返回 0 → 查 SearXNG stderr 含 `CAPTCHA (suspended_time=...)`？含 → 等过期，**禁止重启**。不含 → 假死，重启 |
| **🔥 CAPTCHA 中断协议（v5.5.2 新增）** | ① 本轮命中 CAPTCHA → 等 suspended_time（或最少 5min）→ 重试本轮；② 仍 CAPTCHA → **本轮降级为 Firecrawl**，搜完本轮剩余词；③ 下轮恢复 SearXNG；④ 若连续 2 轮触发中断协议 → 全员切 Firecrawl |
| **失败不纠缠** | 连续 2 轮 CAPTCHA → 切 Firecrawl keyless，不再等 |
| **禁止选项** | `web_search` / `ddgs` / `cn.bing.com` 全部不可用 |
