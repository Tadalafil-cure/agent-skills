# v4.6.6 封版审计发现的系统性陷阱

## 陷阱 1：数据定义扩展后消费者未同步

**模式：** 修改了一个数据结构（如 `INDEX_MAP` 字典、枚举列表），但遗漏了遍历该结构的所有消费者——硬编码循环仍然只迭代原来的子集。

**本次案例（两处同一 bug）：**

| 位置 | 修改 | 遗漏 |
|------|------|------|
| `minute_structure_v2.py:221` | `INDEX_MAP` 加了 `cyb` 和 `kc` | `main()` 中 `for idx in ["sh","sz"]` 硬编码未改 |
| `realtime.py:445` | 无 — 完全独立的遗漏 | `check_minute_structure` 中 `for ... in [("sh","上证"),("sz","深证")]` 未扩展 |

**症状：** 科创50 和创业板的数据文件已生成（`minute_structure_v2_kc.csv` / `minute_structure_v2_cyb.csv`），但盘中简报只输出上证和深证的分钟线，科创50 标注"未覆盖"——实际上数据就在那里，只是没读。

**修复规则：**
1. 修改任何 `INDEX_MAP` / `INDICES` / `SYMBOLS` 类字典后，全局搜索所有遍历该结构的 for 循环
2. 优先用 `for key in DICT:` 而非 `for key in ["hardcoded","list"]` 
3. 修改后跑 `realtime.py --json` 验证分钟线四指数全部有输出

## 陷阱 2：版本号引用不一致

SKILL.md 引用模板 v1.7 但模板实际已升至 v1.8——版本号跨越三个位置（页头、检查清单标题、A21标注、页脚）时容易遗漏。\n解决：单文件内所有版本标注统一 grep→sed 批量替换，不走逐条 patch。\n\n---\n\n## 陷阱 3：跳过市场状态检查直接出报告\n\n**模式：** 盘后走了盘中简报流程，或盘中走了盘后全流程。两种流程的数据源和分析深度完全不同。\n\n**症状：** 盘中简报引述前日报告判断但前日报告不存在；盘后报告用了实时行情代替日线 OHLC。\n\n**修复：** 任何分析触发前，Phase -1b 先跑 `market_status()` 确认状态——收盘→盘后全流程，盘前/开盘/午休→盘中选择。\n\n---\n\n## 陷阱 4：数据管线未验证就出报告\n\n**模式：** 跳过管线逐层验证，直接用未经验证的数据启动 Agent 分析。\n\n**症状：** 日线停在 7/10 而非 7/13（时效检测失效）；分钟线四指数只出了 sh/sz（process.py 硬编码）；diverge=-1 被误读为底背离。一份报告基于三处数据错误，被层层推翻。\n\n**修复：** Phase 1 逐层验证——日线→结构→序列→裁决→分钟线，五层全过才能进 Phase 2。验证脚本见 `references/execution-checklist.md`。\n\n---\n\n## 陷阱 5：A20 违规——主 Agent 直接读博文\n\n**模式：** 主 Agent 在统稿阶段用 `terminal` 直接跑 `blog_monitor.py` 读取博文原文，而非派独立子 Agent。\n\n**症状：** 博客原文的措辞（"钝化级别极小""来啥接啥"）进入主 Agent 上下文→第九节比对变成\"自我验证\"→不是独立验证。\n\n**修复：** 博客 Agent 必须是 `delegate_task` 独立子 Agent，主 Agent 只接收结构化摘要。⛔ 绝对禁止主 Agent 直接运行 blog_monitor.py。\n\n---\n\n## 陷阱 6：diverge 列含义搞反\n\n**模式：** 分钟线 CSV 的 `diverge_60/90/120` 列含义是：`-1`=无背离，`0`=中性，`>0`=底背离。Agent 将 `-1` 误读为"底背离出现"。\n\n**症状：** 报告写\"三周期底部钝化出现\"，实际四指数全无背离。\n\n**修复：** 每次读分钟线 CSV 时显式注释 `diverge=-1=无背离，diverge>0=底背离`，不凭直觉推断。
